# TPC-H Base Implementation - Stage 1: TODO Plan

You are an expert database engineer and skilled C++ programmer.
Write a specialized high-performance TPC-H analytical engine in C++ that is optimized to only execute a predefined set of queries.
The database engine should run the queries described in `$queries_path` ($query_count $query_word).
Datatypes and operators can be hard-coded into the program to avoid interpretation overhead.

## Data Format

Input data is a TPC-H table directory:
- Location: `$data_path`
- Files: `customer.tbl`, `lineitem.tbl`, `nation.tbl`, `orders.tbl`, `part.tbl`, `partsupp.tbl`, `region.tbl`, `supplier.tbl`
- Format: pipe-delimited TPC-H `.tbl` rows, preserving SQL column types declared in the table schema
- Runtime boundary: generated runtime and Dockerized MonetDB baseline must consume the same logical TPC-H rows

## Implementation Files

You need to implement the following source groups:
1. $loader_path: Define `RawData` structure and implement TPC-H table loading (loader skeleton provided)
2. $builder_path: Define `Engine` structure and build indexes from RawData (**start from skeleton**)
3. $query_impl_path: Implement the query dispatcher (thin forwarding in `query_impl.cpp`) and put non-trivial kernel logic into focused query modules created on demand, for example `query_q2.cpp`, `query_q9.cpp`, `query_q12.hpp`, or `query_shared_groupby.hpp`.

**Important**: The build and query templates are minimal skeletons. You need to implement:
- Engine data structures suitable for TPC-H joins, filters, aggregations, ordering, and numeric output
- Query dispatcher and all requested query execution logic, with focused query modules owning split kernels rather than pushing every kernel back into `query_impl.cpp`
- CSV output formatting

The supported TPC-H workload spans Q1-Q22 across scans, multi-table joins, semi/anti joins, correlated subqueries, case/ratio aggregates, string predicates, ordering, and top-k result shapes. Implement the subset requested by this conversation, but keep the dispatcher and execution structure consistent with the Q1-Q22 module registry.

Prefer editing an existing focused `query_q*.cpp` / `query_q*.hpp` module for the active family, or creating a new `query_q*.cpp` / `query_q*.hpp` pair before falling back to a monolithic rewrite of `query_impl.cpp`.
Use `query_shared_*.cpp` / `query_shared_*.hpp` only for pure shared helpers such as timestamp formatting or reusable aggregation kernels; they must never own dispatcher logic, `dispatch_unimplemented_query(...)`, or any `execute_q<N>(...)` entrypoint. The dispatcher stays small and stable.

Do not treat the loader as the primary optimization target for this stage. Focus on getting a working vertical slice through build and query first.

## Query Interface

The query interface uses stdin/stdout protocol:
- Input: Multiple lines of `<QUERY_NR> <QUERY_PARAMETERS>` (e.g., "1" for Q1, "2 'host_0' ..." for Q2)
- Empty line signals end of input
- Output: One CSV file per query: `result<RUN_NR>.csv` (RUN_NR starts from 1)
- Timing: Print `"<QUERY_ID> | Execution ms: <time>"` and `"<QUERY_ID> | Query ms: <time>"` to stdout after each query
- stdout must only contain timing/status lines; do not print query result rows to stdout

Use the header-only C++ parser defined in `$args_path` to parse query requests.

## Output Requirements

Write the output of each query into a separate csv file:
- Name: `result<RUN_NR>.csv` where RUN_NR starts from 1
- CSV format: delimiter=',', escapechar='\\', quotechar='"', header=True
- Column order must match the QueryContract output_columns exactly

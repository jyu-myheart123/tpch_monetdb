Output protocol: do NOT open `result*.csv` directly in query modules. Include `query_impl.hpp`, write the full CSV payload (header + rows) into `get_last_query_result().csv_output`, and set `get_last_query_result().valid = true` only when `should_materialize_query_output()` is true. The dispatcher/runtime owns the physical `result<RUN_NR>.csv` write and will only write it in full_csv mode. Correctness uses full CSV output; printed performance, optimization, final runtime, and speedup use no_output/kernel timing, so query modules must skip expensive CSV row formatting when `should_materialize_query_output()` is false while still reporting kernel timing and row-count metadata when available. Treat CSV string construction as result materialization, not kernel-only compute, when placing timing boundaries. When query-local code can isolate compute from CSV formatting, set `get_last_query_result().has_kernel_ms_override = true` and `get_last_query_result().kernel_ms_override = <kernel-only ms>` before materialization so the dispatcher prints kernel-only `Execution ms`. Do NOT use `printf` or `std::cout` for CSV data rows - stdout is reserved for timing/status lines only.
